 这个仓库里一共有 3 种模式：

  plan    集中式规划
  chat    单次模拟聊天
  dialog  真正多机器人轮流对话

  它们通过参数控制：

  python run_dialog.py --comm_mode plan
  python run_dialog.py --comm_mode chat
  python run_dialog.py --comm_mode dialog

  在代码里选择位置是：

  run_dialog.py -> LLMRunner.__init__()

  核心逻辑：

  if llm_comm_mode in ["plan", "chat"]:
      self.prompter = SingleThreadPrompter(...)
  else:
      self.prompter = DialogPrompter(...)

  所以：

  plan / chat 走 SingleThreadPrompter
  dialog      走 DialogPrompter

  ———

  # 总览对比

  | 模式 | 参数 | 类 | 模型调用方式 | 是否真多轮 | 推荐程度 |
  |---|---|---|---|---|---|
  | 集中式规划 | plan | SingleThreadPrompter | 一次让模型输出全体动作 | 否 | 最推荐 |
  | 单次模拟聊天 | chat | SingleThreadPrompter | 一次让模型模拟讨论并输出全体动作 | 否 | 可尝试 |
  | 多机器人对话 | dialog | DialogPrompter | Alice/Bob/Chad 轮流调用模型 | 是 | 慎用 |

  ———

  # 1. plan：集中式规划

  ## 1.1 它是什么？

  plan 模式就是：

  一个中央 planner 看到全部环境信息，然后直接决定所有机器人的下一步动作。

  它不会让 Alice、Bob、Chad 真正讨论。

  模型扮演的是：

  中央调度员 / 总规划器

  输出类似：

  EXECUTE
  NAME Alice ACTION PICK blue_square PLACE panel2
  NAME Bob ACTION WAIT
  NAME Chad ACTION WAIT

  ———

  ## 1.2 实现流程

  代码位置：

  prompting/plan_prompter.py
  SingleThreadPrompter

  执行链路：

  LLMRunner.one_run()
    ↓
  SingleThreadPrompter.prompt_one_round()
    ↓
  compose_system_prompt()
    ↓
  query_once()
    ↓
  query_ollama_chat()
    ↓
  Ollama llama3.3:70b
    ↓
  返回 EXECUTE 文本
    ↓
  parser.parse()
    ↓
  feedback_manager.give_feedback()
    ↓
  RRT 执行

  ———

  ## 1.3 prompt 里有什么？

  plan 模式的 prompt 由这些东西组成：

  task_desp = env.describe_task_context()
  action_desp = env.get_action_prompt()
  obs_desp = env.describe_obs(obs)
  history_desp = self.compose_round_history()
  feedback = plan_feedbacks
  comm_prompt = get_plan_prompt(env)

  也就是：

  任务规则
  + 动作空间
  + 当前场景描述
  + 历史执行记录
  + 上次失败反馈
  + 让模型集中规划的指令

  其中 get_plan_prompt() 大概要求：

  Reason about the task step-by-step.
  Find the best strategy to coordinate the robots.
  Propose exactly one action per robot.
  Strictly follow Action Output Instruction.

  ———

  ## 1.4 模型调用次数

  每个 high-level step 中，默认最多 replan 5 次：

  --num_replans 5

  但每次 replan 只调用模型一次。

  所以单步最大调用次数：

  plan: 最多 5 次

  通常成功的话只要 1 次。

  ———

  ## 1.5 优点

  ### 优点 1：最稳定

  因为模型直接输出最终格式：

  EXECUTE
  NAME Alice ACTION ...
  NAME Bob ACTION ...
  NAME Chad ACTION ...

  parser 最容易处理。

  ———

  ### 优点 2：速度快

  一次 replan 只调用一次模型。

  对于 llama3.3:70b 这种大模型很重要，因为 70B 推理慢，多调用几次会明显拖慢评估。

  ———

  ### 优点 3：更适合 evaluator

  完整评估最多 30 个 episode，每个 episode 最多 10 个 high-level step。

  如果用 plan，调用次数相对可控。

  ———

  ### 优点 4：反馈闭环清楚

  失败后，程序会把环境反馈直接塞回下一次 prompt：

  IK failed
  Collision detected
  Out of reach
  Task constraint failed

  模型下一次重新规划即可。

  ———

  ## 1.6 缺点

  ### 缺点 1：对模型全局规划能力要求高

  模型必须一次性想清楚：

  谁能 reach 哪个物体
  谁应该 WAIT
  谁应该帮别人转运
  是否会放错目标

  ———

  ### 缺点 2：没有显式多机器人协商

  对于需要 handoff 的复杂任务，模型可能没那么自然地想到：

  Alice 先把物体转移到 panel3
  Bob 再接力

  不过 prompt 里已经写了类似策略，所以一般够用。

  ———

  # 2. chat：单次模拟聊天

  ## 2.1 它是什么？

  chat 看起来像多机器人聊天，但注意：

  它不是程序层面的多轮对话。

  它仍然只调用一次模型。

  区别只是 prompt 风格变成：

  The robots discuss to find the best strategy...
  Their entire discussion and final plan are:

  也就是说，让模型在一次回复里自己模拟 Alice/Bob/Chad 的讨论，然后最后输出 EXECUTE。

  ———

  ## 2.2 实现流程

  chat 和 plan 用的是同一个类：

  SingleThreadPrompter

  区别在：

  if self.comm_mode == "plan":
      comm_prompt = get_plan_prompt(self.env)
  elif self.comm_mode == "chat":
      comm_prompt = get_chat_prompt(self.env)

  所以流程几乎一样：

  LLMRunner.one_run()
    ↓
  SingleThreadPrompter.prompt_one_round()
    ↓
  compose_system_prompt()
    ↓
  get_chat_prompt()
    ↓
  query_once()
    ↓
  Ollama
    ↓
  模型一次性输出讨论 + EXECUTE
    ↓
  parser 只处理 EXECUTE 后面的内容
    ↓
  feedback
    ↓
  RRT 执行

  ———

  ## 2.3 模型可能输出什么？

  例如：

  Alice: I can reach panel1, panel2, panel3. I should move blue_square to panel2.
  Bob: I should wait this round.
  Chad: I should wait this round.

  EXECUTE
  NAME Alice ACTION PICK blue_square PLACE panel2
  NAME Bob ACTION WAIT
  NAME Chad ACTION WAIT

  真正被执行的是：

  EXECUTE
  NAME Alice ACTION ...
  NAME Bob ACTION ...
  NAME Chad ACTION ...

  前面的聊天只是模型自己的推理文本。

  ———

  ## 2.4 模型调用次数

  和 plan 一样：

  每次 replan 调用一次模型

  所以单步最大：

  chat: 最多 5 次

  ———

  ## 2.5 优点

  ### 优点 1：比 plan 更鼓励推理

  它会诱导模型显式分析：

  Alice 能做什么
  Bob 能做什么
  Chad 能做什么
  是否需要协作

  对复杂任务可能有帮助。

  ———

  ### 优点 2：调用成本和 plan 差不多

  虽然输出可能更长，但还是一次模型调用，不像 dialog 要反复调用。

  ———

  ### 优点 3：有助于 handoff 推理

  比如模型可能在讨论中意识到：

  Alice 够不到最终目标，但可以放到中间 panel
  Bob 可以接力

  ———

  ## 2.6 缺点

  ### 缺点 1：输出更啰嗦

  parser 最终只需要：

  EXECUTE ...

  但 chat 模式容易输出大量自然语言。

  如果模型忘了输出 EXECUTE，就会失败。

  ———

  ### 缺点 2：格式风险比 plan 高

  模型可能输出：

  Alice will pick blue_square.
  Bob waits.

  而不是严格的：

  NAME Alice ACTION ...

  虽然现在有清洗逻辑，但仍然比 plan 更容易出格式问题。

  ———

  ### 缺点 3：token 更多，速度略慢

  一次输出包含讨论文本，所以 70B 生成时间可能比 plan 长。

  ———

  # 3. dialog：真正多机器人对话式规划

  ## 3.1 它是什么？

  dialog 是真正的多轮对话。

  程序会分别问：

  Alice
  Bob
  Chad
  Alice
  Bob
  Chad
  ...

  每个机器人看到：

  自己的视角 prompt
  + 当前对话历史
  + 环境反馈

  然后发言。

  直到某个机器人输出：

  EXECUTE
  ...

  程序才拿最终计划去执行。

  ———

  ## 3.2 实现位置

  代码：

  prompting/dialog_prompter.py
  DialogPrompter

  核心函数：

  prompt_one_round()
  prompt_one_dialog_round()
  compose_system_prompt()

  ———

  ## 3.3 执行流程

  整体流程：

  LLMRunner.one_run()
    ↓
  DialogPrompter.prompt_one_round()
    ↓
  for replan in num_replans:
      prompt_one_dialog_round()
        ↓
        while n_calls < max_calls_per_round:
            问 Alice
            保存 Alice 回复
            问 Bob
            保存 Bob 回复
            问 Chad
            保存 Chad 回复
            如果某次回复包含 EXECUTE:
                结束对话
      ↓
      parser.parse(final_response)
      ↓
      feedback_manager.give_feedback()
      ↓
      如果失败，反馈加入下一轮 dialog
    ↓
  RRT 执行

  ———

  ## 3.4 每个机器人收到什么？

  和 plan/chat 不同，dialog 用的是：

  env.get_agent_prompt(obs, agent_name)

  例如 sort 里 Alice 会收到偏个人视角的信息：

  You are robot Alice in front of panel2.
  You are collaborating with Bob and Chad.
  Your goal is to place blue_square on panel2.
  You can only reach panel1, panel2, panel3.
  Current scene:
  blue_square is on ...
  pink_polygon is on ...
  yellow_trapezoid is on ...
  Your gripper is empty...
  Think step-by-step about others' response.

  Bob 收到的是 Bob 视角，Chad 收到的是 Chad 视角。

  ———

  ## 3.5 当前聊天历史怎么加入？

  每个机器人发言后，程序会存：

  agent_responses.append(f"[{agent_name}]:\n{response}")

  下一位机器人会看到：

  [Current Chat]
  [Alice]:
  ...
  [Bob]:
  ...

  如果是下一次 replan，还会看到：

  [Previous Chat]
  ...

  ———

  ## 3.6 什么时候结束对话？

  在 prompt_one_dialog_round() 里：

  if 'EXECUTE' in response:
      if replan_idx > 0 or all([v > 0 for v in num_responses.values()]):
          dialog_done = True
          break

  意思是：

  - 如果第一轮 replan，至少要每个机器人都发过言；
  - 后续 replan 如果出现 EXECUTE 可以更快结束；
  - 最多调用：

  max_calls_per_round = 10

  ———

  ## 3.7 模型调用次数

  默认：

  max_calls_per_round = 10
  num_replans = 5

  所以一个 high-level step 理论最多：

  dialog: 10 × 5 = 50 次模型调用

  而 plan/chat 是：

  最多 5 次模型调用

  差距很大。

  ———

  ## 3.8 优点

  ### 优点 1：更接近多智能体协商

  每个机器人可以从自己的能力出发：

  Alice: 我只能 reach panel1-3
  Bob: 我能接 panel3-5
  Chad: 我能接 panel5-7

  这对接力任务可能有帮助。

  ———

  ### 优点 2：更容易发现别人计划的问题

  prompt 明确要求：

  Carefully check and correct others if they made a mistake.

  例如 Bob 可以指出 Alice 放错目标，Chad 可以指出某个物体他够不到。

  ———

  ### 优点 3：对复杂任务理论上更有解释性

  日志里会保存每个 agent 的发言：

  replan0_call0_agentAlice_*.json
  replan0_call1_agentBob_*.json
  replan0_call2_agentChad_*.json

  便于分析谁提出了错误建议。

  ———

  ## 3.9 缺点

  ### 缺点 1：非常慢

  70B 本地模型多次调用会显著变慢。

  完整 evaluator 下，dialog 很容易把时间放大。

  ———

  ### 缺点 2：更容易超时

  每个 run 有：

  run_timeout = 600 秒

  dialog 一步可能就要很多次模型生成，容易导致 episode 超时。

  ———

  ### 缺点 3：格式风险更高

  多机器人讨论时，模型可能一直输出：

  PROCEED

  或者自然语言讨论，但迟迟不给：

  EXECUTE

  也可能最终 EXECUTE 格式不标准。

  ———

  ### 缺点 4：最终只解析最后一个 response

  程序不是把所有机器人提案自动合并成结构化计划。

  它最终拿的是：

  final_response

  然后：

  parser.parse(obs, final_response)

  所以如果最后一个机器人没有把完整 EXECUTE 写好，前面讨论再好也没用。

  ———

  # 4. 三种模式的共同点

  不管哪种模式，模型输出之后，后处理完全一样：

  模型输出文本
    ↓
  clean_llm_response()
    ↓
  parser.parse()
    ↓
  feedback_manager.give_feedback()
    ↓
  PlannedPathPolicy
    ↓
  MultiArmRRT
    ↓
  env.step()
    ↓
  get_reward_done()

  也就是说，区别只在：

  怎么问模型
  怎么组织 prompt
  调用模型几次

  从 EXECUTE 之后开始，三种模式没有本质区别。

  ———

  # 5. 三种模式的 prompt 差异

  ## plan

  prompt 是全局视角：

  Here is the whole task.
  Here is the whole scene.
  You are a central planner.
  Output exactly one action per robot.

  ———

  ## chat

  prompt 也是全局视角，但要求模型自己模拟讨论：

  The robots discuss to find the best strategy.
  They talk in order [Alice], [Bob], [Chad].
  Once they reach agreement, output EXECUTE.

  但程序只调用一次模型。

  ———

  ## dialog

  prompt 是 agent 视角：

  You are Alice.
  Your goal is ...
  Your reachable panels are ...
  Here is previous/current chat.
  Now your response is:

  然后下一次变成：

  You are Bob.
  ...

  是真正程序层面的多轮。

  ———

  # 6. 成功率角度怎么选？

  ## 最推荐：plan

  原因：

  稳定、快、格式可控、默认 evaluator 使用。

  建议作为主力评测模式。

  命令：

  python run_dialog.py --task sort --num_runs 1 --comm_mode plan --skip_display

  ———

  ## 可尝试：chat

  如果发现 plan 模式在复杂协作上经常想不清楚，可以试 chat。

  命令：

  python run_dialog.py --task sort --num_runs 1 --comm_mode chat --skip_display

  适合观察模型是否能通过自我讨论改善策略。

  ———

  ## 慎用：dialog

  dialog 更适合研究和调试多智能体协商，不太适合直接跑完整 evaluator。

  命令：

  python run_dialog.py --task sort --num_runs 1 --comm_mode dialog --skip_display

  如果用 70B，建议先：

  --tsteps 3
  --num_runs 1

  否则会很慢。

  ———

  # 7. 实战建议

  对于这个项目，我建议策略是：

  ## 第一阶段

  只用：

  --comm_mode plan

  先跑通 6 个任务，调 prompt 和 parser。

  ———

  ## 第二阶段

  对失败最多的任务，尝试：

  --comm_mode chat

  看看是否因为模型没有想清楚协作逻辑。

  ———

  ## 第三阶段

  仅在需要分析多机器人协商时试：

  --comm_mode dialog

  不要一上来用它跑 full evaluate。

  ———

  # 8. 一句话总结

  plan:
  一个中央 planner 一次性决策，最快最稳。

  chat:
  还是一次调用模型，但让模型在一条回复里模拟机器人讨论。

  dialog:
  程序真的让 Alice/Bob/Chad 多轮发言，最像多智能体协作，但最慢、最容易超时、格式风险最高。
