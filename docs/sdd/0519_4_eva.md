评估主要由：

  evaluator.py

  驱动。它不是一个复杂的评测框架，而是一个 批量运行器 + 日志收集器 + 成功率统计器。

  整体链路是：

  python evaluator.py
    ↓
  依次调用 6 个任务的 run_dialog.py
    ↓
  每个任务跑 5 次 episode
    ↓
  读取每次 run 生成的 json
    ↓
  统计 success / timeout / 平均步数
    ↓
  写 summary.json 和 evaluator.log

  ———

  # 2. 每个任务怎么被启动？

  evaluator.py 里每个任务都会调用：

  test_run_dialog(task, num_runs, output_dir)

  比如 sort 会启动一个子进程：

  python run_dialog.py \
    --task sort \
    --run_name runs \
    --data_dir <task_output_dir> \
    --start_id -1 \
    --num_runs 5 \
    --skip_display \
    --tsteps 10 \
    --seed 0 \
    --run_timeout 600

  也就是说：

  - evaluator.py 本身不直接和模型交互；
  - 真正跑仿真和调用大模型的是 run_dialog.py；
  - evaluator.py 只负责批量调用它。

  ———

  # 3. 每个 episode 是怎么评估的？

  在 run_dialog.py 中，每次 episode 由：

  LLMRunner.one_run()

  执行。

  一个 episode 的流程是：

  1. reset MuJoCo 环境
  2. 获取当前 observation
  3. 把 observation 转成文本 prompt
  4. 调用 Ollama / llama3.3:70b
  5. 模型输出高层动作
  6. parser 解析动作
  7. feedback 检查可达性、IK、碰撞、任务规则
  8. RRT 规划机械臂路径
  9. MuJoCo 执行动作
  10. 判断任务是否完成
  11. 如果未完成，进入下一 high-level step
  12. 最多执行 tsteps=10 步
  13. 写出 success / failure json

  其中：

  --tsteps 10

  表示每个 episode 最多允许模型做 10 轮高层规划。

  ———

  # 4. 单个任务怎么算成功？

  成功与否不是 evaluator 判断的，而是每个环境自己的：

  env.get_reward_done(obs)

  决定。

  比如 sort 任务里：

  get_reward_done()

  会检查：

  blue_square 是否在 panel2
  pink_polygon 是否在 panel4
  yellow_trapezoid 是否在 panel6

  如果都满足：

  reward = 1
  done = True

  否则：

  reward = 0
  done = False

  run_dialog.py 最后用：

  success = reward > 0

  写入结果 json：

  {
    "step": 4,
    "success": true,
    "timed_out": false,
    "elapsed_time": 123.45
  }

  不同任务的完成条件不同：

  | 任务 | 完成条件大意 |
  |---|---|
  | sort | 方块放到正确 panel |
  | cabinet | 完成柜门/物品相关目标 |
  | rope | rope 移动到目标状态 |
  | sweep | 扫到目标区域 |
  | sandwich | 按要求完成三明治 |
  | pack | 把物品放入目标容器/区域 |

  具体判断都在：

  rocobench/envs/task_*.py

  里的：

  get_reward_done()

  # 5. 超时机制

  evaluator.py 里每个任务默认 timeout：

  DEFAULT_RUN_TIMEOUTS = {
      "sort": 600,
      "cabinet": 600,
      "rope": 600,
      "sweep": 600,
      "sandwich": 600,
      "pack": 600,
  }

  也就是每个 episode 最多：

  600 秒 = 10 分钟

  传给 run_dialog.py：

  --run_timeout 600

  在 LLMRunner.one_run() 里会检查：

  elapsed_time > self.run_timeout

  如果超时：

  timed_out = True
  success = False

  写入：

  {
    "success": false,
    "timed_out": true
  }

  另外，evaluator.py 对整个子进程也有总 timeout：

  total_timeout = num_runs * run_timeout * 1.2

  比如一个任务 5 次，每次 600 秒：

  5 × 600 × 1.2 = 3600 秒

  也就是一个任务子进程最多跑约 1 小时。

  # 6. evaluator 怎么统计结果？

  每个任务跑完后，evaluator.py 会去找：

  run_0/
  run_1/
  run_2/
  run_3/
  run_4/

  里面的 json 文件：

  all_jsons = glob(os.path.join(run_dir, "*.json"))

  比如：

  steps4_success_True.json
  steps9_success_False.json

  然后读取：

  success = json_data.get('success', False)
  steps = json_data.get('step', 0)
  timed_out = json_data.get('timed_out', False)

  统计：

  success_cnt
  total_cnt
  timeout_cnt
  total_steps

  最后得到每个任务的：

  | 指标 | 含义 |
  |---|---|
  | success_count | 成功次数 |
  | total_count | 实际完成的 episode 数 |
  | success_rate | 成功率 |
  | timeout_count | 超时次数 |
  | avg_steps | 成功 episode 的平均 high-level step 数 |
  | total_time | 该任务总耗时 |
  | returncode | 子进程退出码 |

  该任务子进程的标准输出。

  # 11. 模型在评估中的作用

  评估过程中，每个 high-level step 都会调用模型。

  一次 episode 可能调用多次模型：

  step 0:
    问模型一次或多次，因为有 replan
  step 1:
    问模型一次或多次
  ...
  最多 tsteps=10

  如果一次模型输出无法解析，或者 IK/碰撞失败，会触发 replan：

  --num_replans 5

  所以一次 step 最多可能问模型 5 次。

  完整评估理论最多模型调用次数大约是：

  6 任务 × 5 runs × 10 steps × 5 replans = 1500 次

  实际通常少于这个，因为任务可能提前完成，或者失败后跳过。

  # 12. 评估分数本质是什么？

  这个 evaluator 主要看：

  每个任务 5 次中成功几次

  核心指标是：

  success_rate = success_cnt / total_cnt

  例如：

  sort: 4/5 = 80%
  cabinet: 2/5 = 40%
  ...

  另外辅助指标：

  - timeout 数；
  - 成功 run 的平均步数；
  - 总耗时；
  - 子进程 return code。

  但真正最重要的是：

  success_count / total_count

  不会。evaluate 只是运行代码。

  During grading/testing, evaluator.py will be replaced with the original code.

  意思是官方最终评测可能替换 evaluator.py。
  所以不要依赖修改 evaluator.py 来提高成绩。

  prompting/dialog_prompter.py
  prompting/parser.py
  prompting/feedback.py

  # 14. 简单总结

  评估就是：

  evaluator.py
    依次跑 sort/cabinet/rope/sweep/sandwich/pack
    每个任务跑 5 次 run_dialog.py
    每次 run_dialog.py 让大模型规划，MuJoCo 执行
    环境 get_reward_done() 判断成功
    run_dialog.py 写 success json
    evaluator.py 读取 json 统计成功率

  最关键的成功判断来自各任务环境：

  rocobench/envs/task_*.py::get_reward_done()

  最关键的评估输出是：

  rocobench_runs/output/run_*/summary.json

  最关键的调试日志是：

  tasks/<task>/stdout.log
  tasks/<task>/runs/run_*/step_*/prompts/*.json
