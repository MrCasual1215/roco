  ## 3. 架构核心差异：plan_prompter.py

  这是两个版本最大的区别。

  ### code 的方向：确定性 verification layer

  code/prompting/plan_prompter.py 里面加入了更通用的动作合法性校验：

  _validate_against_legal_actions(...)

  它会检查：

  - 每个机器人是否都有 exactly one action
  - action name 是否合法
  - 是否在当前 legal actions 里面
  - 是否重复执行 forbidden action
  - 是否 PICK 同一个物体
  - 是否 PLACE 到同一个目标
  - 是否超过 max parallel actions
  - 是否调用环境自己的：
      - get_allowed_action_names
      - get_max_parallel_actions
      - verify_plan_semantics
      - get_legal_actions
      - get_recommended_plan

  所以 code 的设计思想是：

  > LLM 可以提议动作，但最终必须通过 deterministic validator。
  > 如果 LLM 不可靠，就用 task-level recommended plan / fallback 来兜底。

  这个比较适合你最近在做的 Sort、Pack、Rope 优化。

  ———

  ### code2 的方向：LLM candidate + verifier + blacklist

  code2/prompting/plan_prompter.py 更大：

  code:  681 lines
  code2: 1155 lines

  code2 有这些机制：

  CandidatePlanner
  VerifiedPlanner
  Verifier
  unresolved_plan_feedbacks
  failed_action_blacklist
  failed_plan_blacklist
  _query_openai_compatible_chat

  它的思路是：

  > 先让 LLM 生成 candidate plan，再让 verifier/corrector 检查和修正，同时把失败动作加入黑名单，避免重复失败。

  优点：

  - 处理复杂语言反馈能力更强。
  - 对 sandwich、pack、sort 等任务可以记住失败历史。
  - 可以减少 LLM 一直重复同一个错误动作的问题。

  缺点：

  - 更依赖 LLM 行为稳定性。
  - prompt 更长，流程更复杂。
  - 和 code 现在的 deterministic verifier 体系不完全兼容。
  - 不适合直接覆盖 code 的 plan_prompter.py。

  ———

  ## 5. parser.py 差异

  ### code 新增了 Pack 的 WAIT retreat

  code 允许 Pack 任务里 WAIT 不是原地等，而是沿 PATH 移动到安全停车位：

  if 'PackGroceryTask' in str(self.env) and len(path_pts) > 0:
      wait_target_pose[:3] = path_pts[-1]

  这个很重要。因为之前 Pack 出问题的核心之一是：

  > 一个机械臂 WAIT，但它手里还拿着 milk / cereal，仍然挡在另一个机械臂的 PLACE 路径上。

  所以 code 允许 WAIT 变成“拿着东西撤退到侧边等待”。

  这是 code 优于 code2 的地方。

  ———

  ### code 放宽了 Sort PICK 姿态

  code 里对 Sort 任务：

  if 'SortOneBlockTask' in str(self.env):
      pick_quat = robot_state.ee_xquat.copy()

  意思是 Sort 里抓 cube 时不强制匹配 cube top site 的 quaternion，而是保持当前 gripper 姿态。

  作用：

  - 减少 Bob/Chad 在 panel3 / panel5 上抓取时的 IK 失败。
  - 对 Sort recovery 很关键。

  code2 没有这个改动。

  ———

  ### code2 有 Cabinet cup/mug 特殊处理

  code2 在 Cabinet 任务中对 cup/mug 的 pick-and-place 做了 split：

  > pick -> preplace -> lower -> release

  这个是为了避免 cup/mug 直接斜着放导致偏位。

  code 里没有这个 Cabinet 特殊逻辑。
  所以如果跑 Cabinet，code2 可能比 code 稳一些。

  ———

  ## 6. Sort 任务差异

  这是 code 当前明显更强的地方。

  ### code2 的 Sort

  已经有一些 safe parallel 和 verification：

  - get_allowed_action_names
  - get_max_parallel_actions
  - _sort_parallel_compatible
  - verify_plan_semantics
  - get_legal_actions
  - get_recommended_plan

  但是它还比较基础：

  - action 只有：

  PICK <object> PLACE <location>
  WAIT

  没有 PLACE-only recovery。

  ———

  ### code 的 Sort

  code 增强了很多：

  #### 1. 新增 PLACE-only recovery

  PLACE <object name> <location>

  用于机器人已经抓着 cube 的情况。

  这能恢复一种常见失败：

  > 机器人已经拿到物体，但 parser/RRT/执行中断后，下一轮不能再 PICK，只能 PLACE 当前手里的物体。

  ———

  #### 2. 新增 IK precheck

  can_agent_pick_cube_ik(...)

  在生成 legal actions 时先轻量测试 IK。
  这样可以减少 LLM 推荐一个理论可达但 IK 实际不收敛的 PICK。

  ———

  #### 3. 新增 same-panel reposition

  _allow_same_panel_reposition(...)

  这是为了解决类似：

  > cube 名义上已经在 panel3，但位置不好，Bob 从 panel3 抓取失败。

  于是允许 Alice/Chad 对同一个 panel 做 reposition，把 cube 放到更标准的位置。

  ———

  #### 4. recommended plan 更强

  code 会枚举 compatible action combinations，优先：

  - final placement
  - recovery placement
  - same-panel reposition
  - safe Alice + Chad parallel move

  所以 code 的 Sort 是当前更推荐保留的版本。

  你之前的实验也符合这个判断：

  sort_verifier_test: 10/10 success
  sort_safe_parallel_test: 7/10 success
  sort_safe_parallel_recovery_test: 9/10 success

  后面又加了 same-panel reposition，应该是针对最后那个失败 run 的继续修复。

  ———

  ## 7. Pack 任务差异

  Pack 两边差别比较大。

  ### code2 的 Pack：尝试并行 PLACE

  code2 的 prompt 更鼓励高空分离通道：

  - Alice 走 left/front corridor
  - Bob 走 back/right corridor
  - 两个 PLACE 可以并行
  - bin slot 要非相邻，XY 距离至少 0.35
  - Bob 避免低空中心路径

  这是高效率方向。

  优点：

  - 如果成功，步数更少。
  - 能同时放两个物体。

  缺点：

  - Pack 的 bin 区域很窄。
  - 两个机械臂同时 PLACE 极易 RRT timeout。
  - 手里拿着 tall object，比如 milk/cereal，碰撞几何很大。
  - LLM 稍微给错 waypoint，就会卡。

  你之前 Pack 的失败正是这一类问题：

  Alice PLACE cereal
  Bob WAIT holding milk
  Collision: cereal-Bob
  RRT timeout

  甚至 Bob WAIT 但手里拿着 milk 仍然是障碍物。

  ———

  ### code 的 Pack：保守串行 PLACE + WAIT retreat

  code 把 Pack 改成更保守的策略：

  - PICK 可以并行
  - PLACE 一次只允许一个机器人
  - 另一个机器人 WAIT，并且可以 retreat 到侧边安全位
  - 使用明确 bin slot，不允许 generic "bin"
  - 检查 occupied slot
  - 推荐计划由 get_recommended_plan 给出

  核心策略：

  如果有机器人 holding item：
      选择一个 holding robot PLACE
      另一个 robot WAIT/retreat

  如果两个都空手：
      可以 PICK，但第一轮避免两个中心高物体一起 PICK

  优点：

  - 成功率更高。
  - 更符合 collision/RRT 的实际能力。
  - 避免两个手臂同时挤进 bin 区。

  缺点：

  - 效率低。
  - PLACE 串行后总步数增加。
  - 如果 WAIT retreat 路径本身规划不好，仍然可能挡路。

  所以 Pack 上：

  - code2 是高效率但风险大。
  - code 是低效率但更稳。

  ———

  ## 8. Rope 任务差异

  ### code2 的 Rope：动态分配 rope end

  code2 的 Rope prompt 强调：

  不要硬编码 Alice/Bob 拿哪一端
  根据 IK 可达性动态分配
  如果 Bob pick rope_back_end 失败，就交换

  这更灵活。

  优点：

  - 如果某个 rope end 对某机器人 IK 不可达，可以换边。
  - 对随机 rope 初始状态更适应。

  缺点：

  - LLM 容易输出不一致计划。
  - 容易混用 PICK / PUT。
  - 容易输出 PLACE / LIFT 这类非法动作。
  - rope 是双臂约束任务，动态分配如果没严格校验，很容易破坏 rope-end distance。

  ———

  ### code 的 Rope：固定阶段策略

  code 改成 deterministic staged policy：

  阶段 1：两端都 PICK
  阶段 2：两端一起 PUT

  并且限制动作：

  PICK / PUT / WAIT
  禁止 LIFT / PLACE / MOVE / LOWER / RAISE / DROP

  还有：

  get_recommended_plan
  verify_plan_semantics
  get_allowed_action_names
  get_max_parallel_actions

  并且把 rope pick target 高度调高：

  ret[2] = np.clip(ret[2] + 0.35, 0.42, 0.54)

  这解决了之前 rope 端点 z 太低导致 IK/碰桌的问题。

  优点：

  - 更符合你要求的“先两端都 PICK，再两端一起 PUT”。
  - 更容易被 verifier 检查。
  - 避免 LLM 乱输出 LIFT/PLACE。

  缺点：

  - 固定 Alice front / Bob back，如果这个分配 IK 不可达，容易卡。
  - 不如 code2 灵活。
  - 后续最好加“固定优先 + IK 失败后 swap”的 hybrid 策略。

  ———

  ## 9. Sweep 任务差异

  code2 的 Sweep prompt 更强。

  code2 加了很多 alignment rule：

  Alice/dustpan 距离 cube < 0.20
  Bob/broom 距离 cube < 0.45
  两者都对齐同一个 cube 才能 SWEEP
  否则必须 MOVE
  DUMP 后不能直接 SWEEP，必须重新 MOVE

  code 里这些规则没有。

  所以如果你要跑 Sweep，code2 的 prompt 可能更稳。
  code 在 Sweep 上可能退化，因为模型可能太早 SWEEP，导致失败。

  ———

  ## 10. Cabinet 任务差异

  Cabinet 差异不大，但 code2 在 parser 里有 cup/mug 特殊 split-place。
  这可能让 Cabinet 里 cup/mug 放置更稳。

  code 没有这个逻辑。
  所以 Cabinet 如果出现 cup/mug 放歪、重新抓取 IK 失败，应该考虑把 code2 的那段 parser 特殊处理移植回来。

  ———

  ## 11. RRT / policy 差异

  rocobench/policy.py 里 timeout 不同：

  code:  timeout = 200
  code2: timeout = 100

  也就是说 code 给 RRT 更多时间，可能更容易找到路径，但单次失败会等更久。
  code2 更快失败，更适合批量跑，但成功率可能略低。

  ———

  ## 12. 总体评价

  ### code 更适合作为当前主线

  原因：

  1. Sort 当前优化更多，效果也更好。
  2. Pack 有 conservative verifier + retreat WAIT。
  3. Rope 符合你要求的“先双 PICK，再双 PUT”。
  4. 有通用 deterministic verification layer。
  5. 默认路径和环境变量更符合当前项目目录。

  ### code2 适合拿来 cherry-pick

  建议从 code2 合并这些：

  1. run_dialog.py 的 --light_output
  2. per-run exception catch
  3. evaluator 的 --llm_source / --light_output
  4. Sweep alignment prompt
  5. Cabinet cup/mug split-place parser
  6. 部分 blacklist / unresolved feedback 思想，但不要直接整文件替换

  ———

  ## 13. 不建议直接替换的地方

  不建议直接用 code2/prompting/plan_prompter.py 覆盖 code/prompting/plan_prompter.py。

  原因：

  - code 现在走 deterministic verifier + task hook。
  - code2 走 CandidatePlanner / VerifiedPlanner / blacklist。
  - 两个体系都改了 prompter 主流程，直接覆盖会冲突。
  - 可能破坏当前 Sort/Pack/Rope 的稳定性。

  更合理的是：

  以 code 为主线
  从 code2 小块 cherry-pick：
      light_output
      exception handling
      Sweep prompt
      Cabinet parser fix
      maybe blacklist idea

  ———

  一句话总结：

  > code 是当前更工程化、更确定性、更适合继续优化 Sort/Pack/Rope 的版本；code2 保留了更多 LLM verifier、轻量实验
  > 和部分任务 prompt 修补，适合作为功能补丁来源，但不适合整体替换 code。
